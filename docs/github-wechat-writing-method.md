# GitHub 热门项目公众号写作方法

## 目标

面向技术读者，把一个 GitHub 热门 AI 项目写成可读、可信、可转发的公众号草稿。重点不是复述 README，而是回答：

- 它解决什么具体问题？
- 为什么现在值得看？
- 技术路线和上手成本是什么？
- 哪些指标只是热度，哪些信息能说明质量？
- 读者下一步应该 Star、试用、二次开发，还是继续观察？

## 推荐文章结构

1. **一句话开场**
   - 说明项目定位和典型使用场景。
   - 不用“神器”“颠覆”“必火”等过度词。

2. **为什么值得看**
   - 展示 stars、forks、今日新增 stars、trending rank、主要语言、topics。
   - 明确这些是热度信号，不等于生产可用性。

3. **项目解决的问题**
   - 用一个具体场景切入，例如 RAG、Agent、模型部署、前端 AI 交互、数据处理、自动化工作流。

4. **核心功能亮点**
   - 拆成 3 到 5 点。
   - 每一点说明“功能是什么、对读者有什么用、从 README/代码/指标看到什么证据”。

5. **快速上手**
   - 给安装命令、最小示例或 README 路径。
   - 如果没有实际跑通，必须写明“基于 README 和仓库信息判断”。

6. **适合谁**
   - 开发者、产品团队、研究人员、内容运营或创业团队等。

7. **限制和风险**
   - license、维护频率、issue 活跃度、文档完整度、生产稳定性、同类替代项目。

8. **项目地址**
   - 放 GitHub 链接。
   - 可以提示读者先看 README、examples、issues 和 release。

## 配图方法

优先级：

1. **GitHub 仓库首页截图**
   - 展示项目名、stars、forks、README 首屏。
   - 用于证明热度和定位。

2. **README 中的架构图、界面图或 demo 图**
   - 必须检查项目 license 和图片来源。
   - 不确定版权时不要直接转载。

3. **自绘流程图**
   - 推荐作为主图或正文图。
   - 结构：输入 -> 核心模块 -> 输出 -> 使用场景。

4. **自制对比表**
   - 字段：功能、适合人群、上手成本、风险点、同类替代。
   - 版权风险最低，适合公众号移动端阅读。

## 表达风格

- 技术判断要具体，不写空泛推荐。
- 热点项目可以写“值得关注”，不要写“必须使用”。
- star/fork 用作趋势信号，不直接等同质量。
- 尽量写短段落和清晰小标题，适配手机阅读。
- 结尾给明确行动建议：收藏、试跑 demo、看 issue、等待稳定版。

## 自动提交边界

当前系统适合自动完成：

- 选择 GitHub 高分项目。
- 生成公众号 Markdown 草稿。
- 生成配图计划和版权提醒。
- 保存草稿记录。

真实推送微信公众号草稿箱前必须补齐：

- `WECHAT_APP_ID`
- `WECHAT_APP_SECRET`
- IP 白名单
- 封面图
- 正文图片上传
- 预览检查
- 标题、摘要、作者和版权复核
