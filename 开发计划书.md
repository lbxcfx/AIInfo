# AI 情报搜索与精选系统开发计划书

版本：v1.0  
日期：2026-05-11  
项目暂定名：AI Intel Radar

## 1. 项目介绍

本项目目标是开发一个可持续追踪 AI 行业动态的信息搜索与精选系统。系统从官方博客、RSS、GitHub、论文平台、Hacker News、中文科技媒体、KOL 等信源持续采集信息，对原始内容进行清洗、去重、AI 相关性过滤、摘要翻译、多维评分、事件聚类和排序精选，最后通过网页、RSS、API 和 Agent Skill 对外提供服务。

项目不是简单的新闻聚合器，而是一个“信源可信度 + AI 语义判断 + 规则排序 + 人工反馈回测”的信息处理系统。核心原则是：模型只做语义理解和维度评分，最终排序、阈值、去重、精选由可审计的代码规则控制。

## 2. 对上一版方案的可行性审议

上一版方案总体可行，技术方向正确，能够覆盖采集、过滤、排序、搜索、日报、RSS/API/Agent 接入等关键功能。但需要做以下修正，避免后续复杂度失控。

### 2.1 可行的部分

- 使用 Next.js + TypeScript 做前端是合理的，适合构建信息流、搜索、后台管理和 API 文档页面。
- 使用 FastAPI 做后端 API 是合理的，生态成熟，自动 OpenAPI 文档适合后续开放 API。
- PostgreSQL + pgvector 适合 MVP 阶段同时存储结构化数据和 embedding，用于事件聚类、相似检索和规则回测。
- Redis + 任务队列适合抓取、LLM 处理、日报生成、索引重建等异步任务。
- Meilisearch 或 Typesense 用于全文搜索是合理的，比直接依赖数据库全文索引更适合前端筛选体验。
- LLM 多供应商抽象是必要的，可以避免被单一模型价格、限流、质量波动绑定。

### 2.2 需要修正的地方

1. 后台任务不能依赖 FastAPI BackgroundTasks  
   FastAPI BackgroundTasks 只适合轻量任务。抓取、网页解析、LLM 调用、embedding、日报生成都应进入独立任务队列。

2. 搜索引擎不要同时引入 Meilisearch 和 Typesense  
   MVP 阶段先选一个，减少运维复杂度。本计划建议优先使用 Meilisearch；如果中文分词或复杂过滤不满足要求，再切到 Typesense 或 OpenSearch。

3. Playwright 不能作为默认抓取方式  
   动态浏览器抓取成本高、失败率高、容易被风控。默认顺序应为 RSS/API/静态 HTML，Playwright 只作为少数动态页面兜底。

4. pgvector 适合 MVP，但不是无限扩展方案  
   若后续条目达到百万级、embedding 检索和聚类压力变大，可迁移到 Qdrant、Milvus 或独立向量服务。第一阶段无需引入，避免过度设计。

5. LLM 不应直接决定“是否精选”  
   LLM 只输出 AI 相关性、分类、摘要、实体、五维评分和理由。最终分数、阈值、精选状态、主条选择全部由代码公式计算。

6. 必须引入规则版本和回测机制  
   否则评分规则很难稳定迭代。每次调整权重、阈值、惩罚项，都要能对历史样本重算并比较效果。

7. 需要从第一天记录成本与失败率  
   LLM 调用成本、每源抓取成功率、解析失败率、队列积压、日报生成耗时，都应进入监控。

## 3. 最终技术栈

### 3.1 前端

- Next.js + TypeScript
- Tailwind CSS
- shadcn/ui
- TanStack Query
- Zustand
- Recharts 或 ECharts
- lucide-react

说明：Next.js 当前官方 App Router 支持 TypeScript、布局、服务端组件等能力，适合信息流和管理后台。前端不承担复杂业务判断，只负责查询、筛选、展示和反馈。

### 3.2 后端 API

- Python 3.12+
- FastAPI
- Pydantic v2
- SQLAlchemy 2.x
- Alembic
- asyncpg
- pytest + httpx + pytest-asyncio/anyio

说明：FastAPI 负责 API、鉴权、后台管理接口、OpenAPI 文档，不直接执行重型任务。SQLAlchemy 2.x 支持 asyncio，但要避免一个 AsyncSession 被多个并发任务共享。

### 3.3 任务队列与调度

推荐 MVP：

- Celery + Redis
- Celery Beat 或 APScheduler 负责周期任务

可选简化方案：

- Dramatiq + Redis

最终建议：先使用 Celery。原因是抓取、重试、延迟任务、定时任务、失败记录和监控生态更成熟，适合后续规模化。

### 3.4 数据库与存储

- PostgreSQL
- pgvector
- Redis
- 本地文件系统或对象存储保存原始 HTML 快照

说明：PostgreSQL 负责主数据、规则版本、评分结果、反馈和日报。pgvector 负责 MVP 阶段 embedding 相似度检索和事件聚类。Redis 负责队列、缓存、限流和临时状态。

### 3.5 搜索

- MVP 选择 Meilisearch

使用场景：

- 关键词搜索
- 时间范围过滤
- 分类过滤
- 信源过滤
- 是否精选过滤
- 分数排序

保留替换点：如果中文搜索、复杂排序或大规模索引不满足要求，后续切换到 Typesense 或 OpenSearch。

- Windows 侧用于浏览器验证、文件查看和必要的编辑操作。

- `F:\AIInfo` 仅作为初始资料目录和 Windows 侧文档查看路径，不再作为依赖安装、构建或服务运行路径。
- 原因：`/mnt/f/AIInfo` 是 Windows 文件系统通过 9p 挂载到 WSL，安装 `node_modules`、Python `.venv`、Next.js 构建和大量小文件 IO 明显变慢，容易导致安装卡住。

## 6. 排序与精选机制

### 6.1 信源分层

- T1：官方博客、官方发布页、论文源、GitHub 官方仓库 Release
- T1.5：官方社交账号、官方论坛、官方开发者社区
- T2：可信 KOL、Newsletter、技术博客、媒体深度报道
- T3：泛科技媒体、转载源、低稳定性来源

### 6.2 LLM 五维评分

LLM 仅输出五个维度，每项 0-100：

- impact_score：影响力
- novelty_score：新颖性
- actionability_score：可操作性
- credibility_score：可信度
- audience_fit_score：目标用户相关性

LLM 不输出最终分，不决定是否精选。

### 6.3 最终分公式

初始公式：

```text
base_score =
  impact_score * 0.24
+ novelty_score * 0.20
+ actionability_score * 0.16
+ credibility_score * 0.18
+ audience_fit_score * 0.22

final_score =
  base_score * source_weight
+ freshness_boost
+ trend_boost
- duplicate_penalty
- marketing_penalty
- low_relevance_penalty
```

初始 source_weight：

- T1：1.25
- T1.5：1.10
- T2：0.95
- T3：0.75

初始精选阈值：

- 模型发布/更新：65
- 产品发布/更新：68
- 行业动态：72
- 论文研究：74
- 技巧与观点：76

阈值必须支持后台配置，并可通过历史数据回测。

## 11. 风险与应对

### 11.1 抓取失败和反爬

应对：

- 优先 RSS/API
- 记录 source_health
- 对失败源降频
- Playwright 只兜底
- 不抓取明确禁止抓取或需要登录的数据

### 11.2 LLM 成本失控

应对：

- 先预筛再评分
- 缓存模型结果
- 按内容 hash 去重
- 模型分层
- 记录 token 成本

### 11.3 排序质量不稳定

应对：

- LLM 不输出最终分
- 规则公式可版本化
- 历史样本回测
- 人工反馈闭环

### 11.4 聚类误合并

应对：

- 限制时间窗口
- 结合实体重叠
- 相似度阈值保守
- 提供人工拆分能力

### 11.5 中文搜索体验不佳

应对：

- 先用 Meilisearch 验证
- 保留搜索引擎抽象
- 必要时切换 Typesense/OpenSearch

## 12. 需要确认的事项

以下事项需要项目负责人确认后才能锁定实现细节：

1. 系统使用范围  
   是内部自用、团队内部使用，还是公开开放？这会影响登录、API Key、限流、风控和部署成本。

2. 模型供应商  
   是否优先使用 OpenAI、DeepSeek、通义千问，还是从第一版就做多供应商切换？

3. 首批信源语言  
   是否中英文同时做？建议先英文官方源 + 少量中文高质量源，避免第一版噪声过大。

4. 是否需要用户系统  
   如果公开开放，必须做账号、API Key、限流和后台权限；如果内部用，第一版可用简单管理员账号。

5. 是否需要推送  
   是否接入飞书、企业微信、Telegram 或邮件日报？

6. 原始网页保存策略  
   是否保存完整 HTML 快照？保存快照利于排错和回溯，但会增加存储成本。

7. 是否允许使用 RSSHub  
   RSSHub 对补齐信源很有价值，但需要确认部署方式和维护责任。

### 13.5 大模型与向量模型使用方案

第一版不使用 GPT 模型，统一使用智谱 GLM 与智谱 `embedding-3`。智谱提供 OpenAI API 兼容接口，后端可以使用 OpenAI Python SDK 的 `chat.completions` 和 `embeddings` 调用方式，但 provider、base_url、模型名都按智谱配置。

智谱 OpenAI 兼容接入要点：

- Python SDK：`openai>=1.0`
- API Key 环境变量：`ZAI_API_KEY`
- Base URL：`https://open.bigmodel.cn/api/paas/v4/`
- 对话接口：`client.chat.completions.create(...)`
- 向量接口：`client.embeddings.create(...)`
- 向量模型：`embedding-3`

模型分层原则：

- `glm-4.5-air`：最低成本，适合高吞吐、低风险、可容忍轻微误差的任务。
- `glm-5-turbo`：平衡质量和成本，适合摘要、分类、实体、五维评分等主流程。
- `glm-5.1`：最高质量，适合高价值信息复核、规则审计、回测抽样和复杂争议样本。

第一版建议使用可切换模型配置，不把模型名称写死在业务代码中。

环境变量：

```text
LLM_PROVIDER=bigmodel
ZAI_API_KEY=
BIGMODEL_BASE_URL=https://open.bigmodel.cn/api/paas/v4/
LLM_MODEL_RELEVANCE=glm-4.5-air
LLM_MODEL_TRANSLATION=glm-4.5-air
LLM_MODEL_SUMMARY=glm-5-turbo
LLM_MODEL_CLASSIFICATION=glm-5-turbo
LLM_MODEL_ENTITY=glm-5-turbo
LLM_MODEL_SCORING=glm-5-turbo
LLM_MODEL_AUDIT=glm-5.1
EMBEDDING_PROVIDER=bigmodel
EMBEDDING_MODEL=embedding-3
EMBEDDING_DIMENSIONS=1024
LLM_MAX_TOKENS_RELEVANCE=1024
LLM_MAX_TOKENS_SUMMARY=3000
LLM_MAX_TOKENS_SCORING=3000
```

默认分工：

| 任务 | 默认模型 | 说明 |
| --- | --- | --- |
| AI 相关性预筛 | `glm-4.5-air` | 低成本、高吞吐，目标是高召回 |
| 标题翻译 | `glm-4.5-air` | 标题翻译成本低，必要时失败重试到 `glm-5-turbo` |
| 摘要生成 | `glm-5-turbo` | 生成中文短摘要和长摘要 |
| 分类 | `glm-5-turbo` | 五大分类，要求输出稳定 JSON |
| 实体抽取 | `glm-5-turbo` | 公司、产品、模型、人物、论文、GitHub 项目 |
| 五维评分 | `glm-5-turbo` | 输出五个维度分，不输出最终分 |
| 规则审计/回测抽查 | `glm-5.1` | 仅用于少量高价值样本复核 |
| embedding | `embedding-3` | 聚类和相似检索，默认 1024 维 |

可选升级：

- 如果聚类质量不足，将 `embedding-3` 维度从 1024 提升到 2048。
- 如果向量存储和检索压力较大，将 `embedding-3` 维度降到 512，并重新建库与回测。
- 如果评分波动较大，对 T1 高价值信息使用 `glm-5.1` 二次评分。
- 如果成本过高，优先扩大 `glm-4.5-air` 的使用范围，但必须经过同一批标注样本评估。

向量模型配置：

- 模型：`embedding-3`
- 默认维度：1024
- 可选维度：256、512、1024、2048
- 单条请求输入数组最大 64 条文本
- 单条文本上限按智谱文档控制在 3072 tokens 以内
- pgvector 字段维度必须与 `EMBEDDING_DIMENSIONS` 一致；上线后变更维度需要重建向量列和索引。

调用接口建议：

- 文本生成、分类、评分统一使用智谱 OpenAI 兼容 `chat.completions`。
- 向量化统一使用智谱 OpenAI 兼容 `/embeddings`。
- 结构化任务要求输出 JSON，并在代码层做 JSON Schema 校验与失败重试。
- GLM 模型会产生 reasoning_content，`max_tokens` 太低时可能 `finish_reason=length` 且正式 `content` 为空；生产默认值应给预筛 1024、摘要/评分 3000 以上的输出预算，并监控截断率。
- 每次调用保存 prompt_version、model、input_hash、output_json、token_usage、latency_ms、cost_estimate

### 13.6 开发启动结论

按当前计划可以启动开发，但推荐执行顺序如下：

1. 先完成阶段 0-3：项目骨架、数据库、采集、标准化、去重。
2. 同时准备 100-300 条人工标注样本，用于预筛和排序回测。
3. 阶段 4 开始接入 LLMProvider，但业务代码只依赖抽象接口。
4. 阶段 6 之前必须确定第一版 source_weight、分类阈值和回测指标。
5. 阶段 9 之后启动后台服务和前端页面，进行真实浏览器验证。

## 14. 推荐实施结论

方案可行，但必须按 MVP 控制范围：先做 30-50 个高质量信源，优先跑通“采集、清洗、预筛、摘要、五维评分、规则精选、前端展示”这条主链路。不要第一版就追求过多信源、过多模型、过多推送渠道。

最终技术栈建议锁定为：

- 前端：Next.js + TypeScript + Tailwind + shadcn/ui
- 后端：FastAPI + SQLAlchemy 2.x + PostgreSQL
- 队列：Celery + Redis
- 向量：pgvector
- 搜索：Meilisearch
- 抓取：httpx + feedparser + trafilatura，Playwright 兜底
- AI：LLMProvider 抽象，多模型可切换
- 本地开发：Windows + WSL2 Ubuntu，不使用 Docker
- 生产部署：Linux VM/systemd 起步，Docker 作为后续可选项

第一版成败关键不在 UI，而在信息处理机制是否可控。最重要的工程边界是：模型负责理解，代码负责裁决。

### 2026-05-12 阶段 4.6 GitHub 官方推荐源与重评分

调研结论：

- GitHub 官方有 Trending 页面：`https://github.com/trending`，但没有公开 REST Trending API。
- GitHub 官方 Topic 页面可作为专题候选来源，例如 `https://github.com/topics/artificial-intelligence`。
- GitHub REST API 的稳定结构化入口是 repository search 和 repository metadata。
- 因此采集策略调整为：
  - GitHub 官方 Trending/Topic 页面负责发现候选项目。
  - GitHub REST API 负责补充 stars、forks、language、topics、pushed_at 等结构化指标。

已完成：

- 新增 GitHub 官方来源：
  - `GitHub 官方 Trending`
  - `GitHub AI 官方 Topic`
- 原 `GitHub AI 趋势` 旧规则源停用，原因：
  - 旧规则按 `updated` 拉取，容易把 star/fork 很低但近期更新的项目排到前面。
- `GitHub AI 热门仓库` 改为更严格查询：
  - `stars > 1000 / forks > 50`
  - `stars > 500 / forks > 30`
  - `stars > 300 / forks > 20`
  - 默认按 `stars` 排序，而不是按 `updated` 排序。
- GitHub 项目评分加入硬性约束：
  - `stars < 200` 且 `forks < 30`，除非有很强 stars_today 信号，否则最高分封顶 65。
  - `stars < 500` 且 `forks < 50`，最高分封顶 72。
  - 高 star、高 fork、官方 Trending 靠前、stars_today 高的项目才允许进入高分区。
- 新增 `/api/v1/admin/items/rescore-github`，用于对历史 GitHub 数据重新评分并重建索引。
- GitHub 官方页解析器增加导航链接过滤，避免把 `sponsors/explore` 等非仓库链接当作仓库。
- GitHub API 403/404/429 单仓库容错，不再导致全局采集失败。
